# 2부 모델 주도 설계의 기본 요소
## 4. 도메인 격리
도메인과 도메인과 관련 없는 것을 구분한다.
### 계층형 아키텍처
도메인에 관계있는 코드와 관계없는 코드가 섞이면 나오는 단점
* 도메인 관련 코드를 확인하고 분석하기 힘들어진다. 예를들어 업무 규칙을 수정하려는데, UI 코드부터 확인해야 한다던지 DB 코드를 봐야 한다던지..
* 자동화 테스트 불가 : 로직과 기술이 섞여있으면 자동화 테스트가 어렵다.
> 업무 규칙을 변경하고자 UI 코드나 데이터베이스 코드, 또는 다른 프로그램 요소를 세심하게 추적해야 할지도 모른다. 응집력있고, 모델 주도적인 객체를 구현하는 것이 비현실적인 이야기가 돼버리고 자동화 테스트가 어려워진다. 기술과 로직이 모두 각 활동에 포함돼 있다면 프로그램을 매우 단순하게 유지해야 하며, 그렇지 않으면 프로그램을 이해하기가 불가능해진다.

계층형 아키텍쳐는 보통 4가지 계층으로 나누고, 각 계층은 같은 계층의 다른요소나 아래 위치한 계층에만 의존하는것이 일반적이다. 도메인 모델과 관련된 코드는 도메인 계층에 위치하되 다른 계층의 코드와는 분리시킨다.
> 계층화의 핵심 원칙은 한 계층의 모든 요소는 오직 같은 계층에 존재하는 다른 요소나 계층상 “아래” 에 위치한 요소에만 의존한다는 것이다.
> …
> 도메인 모델과 관련된 코드는 모두 한 계층에 모으고 사용자 인터페이스 코드나 애플리케이션 코드, 인프라 스트럭처 코드와 격리하라


# 4부 전략적 설계
## 14. 모델의 무결성 유지
* 대규모 시스템에서 하나의 도메인 모델을 완전하게 단일화하는 것을 생각할 수도 있지만, 이는 현실적으로 비용(개발, 시간)을 고려해보면 실현 불가능하다.

> 용어가 모호하지 않고 모순되는 규칙이 없는 모델의 내적 일관성을 단일화(unification) 이라고 한다.
> …
> 이상적인 세계라면 기업의 전체 도메인을 아우르는 단일 모델이 있을 것이다.
> …
> 전사적 시스템 차원에서 단일화를 유지하기란 생각보다 복잡하다.
> …
> 우리에겐 모델의 주요 부분을 밀접하게 단일화 하는 수단이 필요하다. 이것은 저절로 되거나 선의의 의도가 있다고 해서 되는 것이 아니며, 의식적인 설계 결정과 구체적인 프로세스를 거쳐서만 달성될 수 있다.

* 전사적 단일화된 모델이 실현 불가능하더라도 어떤것을 단일화할지, 하지 않을지를 결정할 필요는 있다.
> 어떤것을 단일화 해야 하는 지에 관한 적극적인 의사결정과 어떤것을 단일화해서는 안되는 지를 실용적으로 인식하는 안목이 결합되면 현재 처한 상황을 명학하게 나타내고 다른 이와 공유할 수 있는 형상을 만들어낼 수 있다.

* 결국 모델간의 경계를 표시해줘야 하는데, 모델의 단일화를 지키는 범위를 컨텍스트(Context)라고 하고, 이 컨텍스트 간의 관계는 컨텍스트 맵(Context Map), 모델의 적용 가능성 범위, 즉 경계를 바운디드 컨텍스트(Bounded Context)로 정의한다.
